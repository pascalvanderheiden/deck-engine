- **Shadcn Theme Rendering Bug Fixed (2026-03-14):** Three bugs caused all themes to look visually identical: (1) APP_JSX template never destructured `theme` from project config or passed it to `<SlideProvider>`, so `data-theme` always defaulted to "dark". (2) COVER_SLIDE_CSS template hardcoded dark-theme rgba values for meta bar (`rgba(22,27,34,0.8)`) and orb gradients — replaced with `var(--surface-overlay)` and `color-mix(in srgb, var(--token) N%, transparent)`. (3) No `color-scheme` CSS property on any theme — added `color-scheme: light` to shadcn/light, `color-scheme: dark` to dark. The actual CSS cascade was correct (unlayered theme `:root` properly overrides `@layer base` defaults), but the scaffolded project's hardcoded colors and missing theme prop made themes indistinguishable visually. Fixed in both engine source and test project. 136 tests pass.

- **Theme Creator Engine Design Perspective (2026-03-16):** Full analysis of requirements for an on-demand theme generation system. Classified all ~95 tokens into 3 tiers: Tier 1 (32 tokens, MUST define — visual breakage if missing), Tier 2 (24 tokens, SHOULD define — generic without them), Tier 3 (~39 tokens, MAY inherit from base). Defined 10 non-negotiables for the engine: complete Tier 1 coverage, auto-generated @theme inline bridge, derived overlay/glow tokens (not AI-chosen independently), WCAG AA contrast validation, color-scheme/luminance match, atomic CSS+descriptor generation, strict file structure, no hardcoded hex in component CSS, theme loader registration, single unlayered :root block. Identified 6 major risks: color soup (no 60/30/10 ratio), token orphans (mismatched glow/overlay colors), typography disasters (display fonts at body sizes), broken Tailwind bridge, design system mismatch, stale descriptors. Defined minimum viable descriptor schema (9 required sections). Written to `.squad/decisions/inbox/saul-theme-creator-design-perspective.md`.
- **Theme Creator Architecture Finalized (2026-03-14):** Final architecture locked after design meeting synthesis. Rusty incorporated all three positions (Saul's token-tier framework, Basher's package boundary and file pre-generation strategy, owner directives on user scope and expressiveness). ARCH-004 filed in decisions.md. Key wins: (1) Token tier model accepted — Tier 1 (required identity), Tier 2 (derived/generated), Tier 3 (optional/inherited) — prevents both expressiveness-killing conservatism and unsolvable incoherence. (2) "Rule 2 — Exported CSS is always canonical and complete" — compromise on payload: recipe may omit structural tokens, but final CSS must write full surface. (3) 10 design non-negotiables codified (complete Tier 1 coverage, WCAG AA on critical pairs, atomic CSS+descriptor, single unlayered :root, etc.). (4) 6 failure modes documented with mitigations (color soup, token orphans, typography disasters, Tailwind bridge breaks, design system mismatch, stale descriptors). (5) v1 user: monorepo maintainers, not mass market. v1 output: local decks. v1 expressiveness: full range (corporate through funky-punk). v1 preview: mandatory canonical 6-slide build + screenshots as release gate. Canonical preview deck exercises: title/hero, content-frame, cards/grid, callout, image-led, closing/CTA. Architecture now ready for implementation planning (8-phase rollout starting with token contract formalization).
