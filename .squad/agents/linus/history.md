- **Project:** DECKIO / deck-engine — a presentation engine and scaffolder for creating beautiful, Copilot-powered slide decks
- **Stack:** Node.js monorepo (npm workspaces), Vite, HTML/CSS/JS, packages: `@deckio/deck-engine` and `@deckio/create-deck-project`
- **Created:** 2026-03-13

## Learnings

<!-- Append new learnings below. Each entry is something lasting about the project. -->
- **2026-03-13 Team Kickoff:** Zero tests in repository; no Vitest, Jest, or Playwright configured. `package.json` missing `test` script. Publishing to npm with zero automated gates—critical Phase 1 blocker. Core untested components: SlideProvider, Slide, Navigation, export logic, CLI scaffolder.
- **Team Findings (Rusty synthesis):** Engine architecture solid: SlideProvider owns state/persistence, Slide handles render states, Navigation owns chrome. Export is screenshot-first (modern-screenshot + Puppeteer). Vite integration critical (dedupes react/react-dom). CLI works but no error handling.
- **Phase 1 Roadmap:** Install Vitest, write 20-30 unit tests for core components, set up GitHub Actions CI gates before npm publish. Phase 1 concurrent with Basher (dev scripts). Phase 2 integration tests + layout tests. Phase 3 E2E deck authoring + export.
- **Phase 1 Complete — Test Foundation Laid:** Vitest 4.x installed at root with workspace support (`vitest.workspace.js`). 33 tests across 3 files, all passing. Test coverage: `deckPlugin()` Vite plugin (6 tests), scaffolder pure functions via extracted `utils.mjs` (23 tests — slugify, packageJson, deckConfig), scaffolder smoke tests (4 tests — bin config, file existence, export validation). Each package runs tests independently with `npx vitest run`. Root `npm test` runs all workspace tests.
- **Architecture Decision — Extracted `utils.mjs`:** Moved `slugify`, `packageJson`, `deckConfig` from `create-deckio/index.mjs` into `create-deckio/utils.mjs` for testability. The CLI entry point auto-runs `main()` on import, making internal functions untestable without extraction. This pattern should be followed for future scaffolder utilities.
- **Key Paths:** Tests live in `packages/<pkg>/__tests__/*.test.js`. Vitest config per-package at `packages/<pkg>/vitest.config.js`. Workspace config at root `vitest.workspace.js`.
- **What's NOT tested yet:** React components (SlideProvider, Slide, Navigation, BottomBar) — need jsdom + React test renderer. Puppeteer export scripts (export-pdf, capture-screen) — need integration test approach. `init-project.mjs` — filesystem-heavy, needs temp dir fixtures. `getStoredSlide()` in SlideContext.jsx — pure function but not exported.
- **Team Update (2026-03-13T00:58:09Z):** Basher's `prepublishOnly` gate is now functional — `npm publish` on deck-engine will run `npm test`. DEV-001 + TEST-001 filed in decisions.md. Saul's audit confirms no test gaps in CSS (no CSS logic). Phase 2 React component tests will require `@vitejs/plugin-react` + jsdom.
- **Theme System Tests (2026-03-13T02:30:00Z):** Added 45 tests for Basher's theme system (49→94 total). Coverage: `resolveTheme()` per-theme resolution + edge cases (empty string, null, unknown theme, custom .css path), `getAvailableThemes()` filtering, theme CSS file structure validation (existence, non-empty, required tokens `--background`/`--foreground`/`--primary`, `:root` block, `@import "tailwindcss"`, `@theme inline` bridge, balanced braces), scaffolder `mainJsx()` theme integration (extracted from `index.mjs` to `utils.mjs` per Architecture Decision), `deckPlugins()` option passthrough, and `tailwindPlugin()` config. Key finding: `resolveTheme("unknown")` silently returns a non-existent path — no runtime validation. This is by design (allows custom themes) but consumers should check `existsSync()` if needed.
- **E2E Scaffolder Verification (2026-03-14):** Full end-to-end test of all 3 theme paths (dark, light, shadcn). Results: All 94 unit tests pass. Scaffolder generates correct files for all themes — vite.config.js, main.jsx, deck.config.js, package.json, slide files all present and correct. `deckPlugin` and `tailwindPlugin` exports from `vite.js` resolve correctly (the original `tailwindPlugin` import bug is FIXED). All theme CSS files (dark.css, light.css, shadcn.css) and global.css resolve. Dev server boots on all 3 themes (Vite ready in ~600-770ms). **BUG FOUND:** Production build (`vite build`) fails on ALL themes — `index.js` re-exports `resolveTheme`/`getAvailableThemes` from `theme-loader.js` which imports Node.js builtins (`fs`, `path`, `url`). Rollup can't bundle these for the browser. Root cause: line 6 of `packages/deck-engine/index.js`. Fix: move theme-loader exports to a separate server-only entry point, or mark them as external in the Vite config. Dev server unaffected (Vite serves modules unbundled in dev).
- **E2E Lesson:** Unit tests testing pure functions (slugify, packageJson, deckConfig) passed while the actual scaffolded project had a production build bug. This proves Ali's point — unit tests alone are insufficient. E2E verification that scaffolds a real project and runs both `dev` and `build` is essential before shipping scaffolder changes.
- **E2E Verification — shadcn adapter + @layer CSS (2026-03-14):** Full matrix verified. 130 unit tests pass. Scaffolder produces correct output for all 3 themes (dark, light, shadcn). shadcn design system adapter works: `components.json`, `src/lib/utils.js`, `jsconfig.json`, `@/` alias in vite.config.js, and `designSystem: 'shadcn'` in deck.config.js all present. `npx shadcn add button` installs `button.jsx` to `src/components/ui/` and project still builds after. Dark/light themes correctly omit all shadcn artifacts. Dev server returns HTTP 200 on all 3 themes (~450-550ms boot). **Production builds pass on all 3 themes** — the old `index.js` re-export bug (PROCESS-001) is fixed; `index.js` no longer imports `theme-loader.js`. @layer CSS strategy verified: `global.css` declares `@layer theme, base, components, utilities;`, tokens + reset in `@layer base`, engine chrome in `@layer components`. Production CSS bundle preserves the layer order. Tailwind utilities correctly land in `@layer utilities` and override engine component styles. No cascade conflicts between engine CSS and shadcn/Tailwind utilities. CLI now uses `@clack/prompts` for interactive prompts (replaces old `fmt` ANSI helpers).
